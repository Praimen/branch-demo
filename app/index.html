Quantum Branch WorkFlow

Goals

  • To establish a clear code base state that is more reflective of the code base in PROD
  • To have more isolated feature branches that can more readily lend themselves to our agile workflow philosophy (shippable stories)
  • To have more confidence about what code is being merged and the likelihood of it being apart of the PROD codebase
  • To establish clear roles and workflow practices around the branches

Pros

  • Push QA to be more of a release/ pre-prod testing environment
  • more predictable code merges
  • clear recovery/revert path
  • more communication needed

Con

  • more branches/complex process
  • Branch Manager needed
  • more communication needed

Feature Branch WorkFlow

How to Feature Branch

  • build the feature branch from the release branch
  • The only branch that should be pulled into your feature branch is the release branch
  • The feature branch should only contain your feature
  • The feature branch should be successfully built before integration build and testing

Working with other features

  • After you have completed a successful feature build , then do an integration build (hopefully this step can be automated and consolidated into a feature/integration build)
  • If an integration build issues arise, consider creating a Co-op branch to resolve issues, but strongly consider sending your successfully isolated feature branch to QA for testing
  • When your feature is possibly dependant on another feature, but a scenario exists in which your feature could go to PROD on its own
  • Always compare feature branch contents vs the release branch

Release Branch WorkFlow

Monday

  • the release branch is created from master
  • all feature branches will pull the release into their branch

During the week

  • As a feature if ready for QA a feature build is done
  • a pull request to the release branch should be created once a feature is ready for QA
  • QA Lead should be a reviewer on the pull request, they will mark on the pull request that a test cycle is beginning on the feature once they have deployed the file
  • QA Lead approval will ensure that the feature has been tested and that there are no other unknown commit/changes between the beginning and ending of the testing commits past her approval
  • The release branch will only contain features that have been tested, have little chance for change, and slated to go to PROD on Sunday or Thursday
  • Developers with be notified when a feature is merged into the release branch

Before Deployment

  • Wednesday(or so) morning the release features going into PROD on Sunday will be merged into
  • This branch will be build and sent to QA for a final production integration testing pass, this will be the build that is send to PROD

Monday after deployment

  • The release branch is verified with the deployments that actually went to production
  • The release branch is merged into master
  • All ongoing feature branches will pull the release into their branch

Co-op Branch WorkFlow

How to Co-Op

  • build the co-op branch from the release branch
  • merge your feature into the co-op branch, then merge any other feature needed into the Co-Op
  • Co-Op testing/integration should not necessarily delay QA testing

When to Co-Op

  • When there is an issue in the Integration build but maybe not in your feature branch, co-op will allow you to troubleshoot particular features with your own
  • when your feature is possibly dependant on another feature, but a scenario exists in which your feature could go to PROD on its own

Int BranchWorkFlow

Integration Branch WorkFlow

  • building to integration
  • merge your feature into the co-op branch, then merge any other feature needed into the Co-Op
  • Co-Op testing/integration should not necessarily delay QA testing

When to Integrate

  • An integration build should be done soon after a feature build that is ready for QA
  • Integration gives you a chance to see your feature co-mingled with other features that have been approved by developers
  • If integration issues arise evaluate the issue to see if it should hamper the QA workflow process
